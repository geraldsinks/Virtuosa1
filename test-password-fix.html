Password Reset Login Issue - FIXED

Problem Identified

The login failure after password reset was caused by bcrypt salt rounds inconsistency:

  • User Model (signup): Used 12 salt rounds
  • Reset Password: Used 10 salt rounds
  • Result: Passwords hashed with different strengths couldn't be compared properly

Fix Applied

FIXED: Updated reset password endpoint to use 12 salt rounds consistently:

// Before (BROKEN) user.password = await bcrypt.hash(password, 10); // After (FIXED) user.password = await bcrypt.hash(password, 12);

Now both signup and password reset use the same bcrypt configuration (12 salt rounds).

Test the Fix

To test the complete password reset flow:

  1. Go to Login Page
  2. Click "Forgot Password?"
  3. Enter your email and request reset
  4. Click the reset link in your email (now points to dedicated reset page)
  5. Set a new password
  6. Try logging in with the new password

Expected Result: Login should now work successfully!

Technical Details

What was happening:

  • Signup: User model pre-save hook hashes password with 12 salt rounds
  • Reset: Direct hash with 10 salt rounds
  • Login: bcrypt.compare() expects consistent hash format
  • Different salt rounds = different hash formats = comparison failure

Files Modified:

  • server/server.js - Line 3064: Updated bcrypt.hash from 10 to 12 salt rounds
  • client/pages/reset-password.html - New dedicated reset page (previous fix)

bcrypt Salt Rounds:

  • 10 rounds: ~2^10 = 1,024 iterations (faster, less secure)
  • 12 rounds: ~2^12 = 4,096 iterations (slower, more secure)
  • Consistency is key for password comparison

Verification

The fix ensures:

  • Consistent hashing across signup and password reset
  • Proper password comparison during login
  • Security maintained with 12 salt rounds
  • Better UX with dedicated reset page
